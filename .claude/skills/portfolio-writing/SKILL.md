# Portfolio Writing — 포트폴리오 기술 문서 작성

## Overview

면접관이 읽는 문서. 첫 장에서 임팩트를 보여주고, 핵심 3~4개 주제에 대해 "어떤 문제를 만났고, 어떻게 접근했고, 무엇이 어려웠고, 어떻게 해결했는가"를 구조적으로 보여준다. 엔지니어링 관점은 라벨 없이 서술 속에 자연스럽게 녹인다.

## When to Use

- 포트폴리오 프로젝트 기술서
- 경력기술서의 프로젝트 상세
- content/portfolio/ 경로의 문서 작성/수정 시

**When NOT to use:**
- 블로그 포스트 → `blog-writing` 스킬 사용
- 사내 기술문서, API 문서

## 첫 장 구조 — 임팩트 퍼스트

첫 장은 면접관이 "이 프로젝트 더 읽어볼 만하다"고 판단하는 페이지. 프로젝트 유형에 따라 구조가 다르다.

### 회사 프로젝트

```
1. 프로젝트 소개 (한 줄 — 무엇인가)
2. 문제 (기존에 무엇이 안 되고 있었는가 — 구체적 수치)
3. 내가 한 일 (어떻게 해결했는가 — 핵심 3~4개만)
4. 비즈니스 기여 (해결 결과가 회사에 어떤 임팩트를 줬는가 — 수치)
```

```
# 예시
무인 증명사진 키오스크 시스템. 촬영-보정-결제-인쇄 전체 파이프라인을 Unity 기반으로 구현.

기존 외주 솔루션은 코어가 소스 없는 C++ DLL로 잠겨 있었고,
하나를 고치면 다른 곳이 깨지는 구조. 로깅 부재로 현장 장애 원인 파악에
수일 소요. CS 업무 과중, 고객 이탈 반복.

아키텍처를 처음부터 새로 설계. 13개 런타임 시스템을 단일 Facade로 통합,
프린터 3종 SDK를 Hybrid Architecture로 추상화, GPU 셰이더 기반 이미지
보정 파이프라인 직접 구현.

적용 후 CS 건수 60% 감소. 버그 수정 사이클: 기존 며칠 → 당일 배포.
```

### 개인 프로젝트

```
1. 동기 (무엇을 하고 싶었는가 — 해결하려는 문제나 만들고 싶었던 것)
2. 접근 (어떻게 어프로치했는가 — 기술 선택과 그 이유)
3. 난관 (과정에서 발생한 핵심 문제들)
4. 해결 (어떻게 풀었는가)
5. 회고 (돌아보면 무엇을 다르게 할 것인가, 현재 상태)
```

## Core Principles

### 1. 톤과 어미

블로그의 "~했다" 체도, 딱딱한 "~적용" 체언 종결도 아닌, **간결한 서술체**를 사용한다.

```
# 금지 (블로그 톤)
"프린터 3종을 통합하면서 많은 걸 배웠다."

# 금지 (딱딱한 체언 종결만)
"프린터 3종 Native SDK 통합. SDK 특성별 Hybrid Architecture 설계."

# 대신 (간결한 서술체 — 자연스럽되 군더더기 없는)
"프린터 3종의 Native SDK를 통합했고, SDK마다 인쇄 안정성이 달라
 상태 조회와 인쇄 경로를 분리하는 Hybrid Architecture를 설계했다."
```

체언 종결("~적용", "~설계")과 간결 서술체("~했고", "~구조다")를 섞어서 자연스러운 리듬을 만든다. 한 가지 어미만 반복하면 기계적으로 보인다.

### 2. 관점 — 라벨 없이 자연스럽게

"사견:", "관점:" 같은 라벨을 달지 않는다. 기술적 판단을 서술하는 흐름 속에서 자연스럽게 엔지니어링 철학이 드러나야 한다.

```
# 금지 (명시적 라벨)
"사견: 추상화는 공통화가 아니라 특수성의 존중이다."

# 금지 (뻔한 감상)
"이 과정에서 아키텍처의 중요성을 깨달았다."
"많은 것을 배울 수 있었다."

# 대신 (판단 근거를 서술하면서 관점이 자연스럽게 드러남)
"3종 프린터에 동일한 인쇄 방식을 강제하면 특정 모델에서 조용히 실패한다.
 추상화의 핵심은 공통 인터페이스를 만드는 것이 아니라, 각 구현체의
 특수한 실패 모드를 얼마나 존중할 수 있느냐에 있다.
 이 판단을 근거로 상태 조회와 인쇄 경로를 분리하는 Hybrid 구조를 택했다."
```

관점이란 **구체적 경험에서 도출된 기술적 원칙이나 판단 기준**이다. "배웠다"가 아니라, "이런 경험을 근거로, 이런 판단 기준을 적용한다"는 것.

### 3. 문서 구조: 3~4개 핵심 주제

전체 시스템을 나열하지 않는다. 면접관이 이 사람의 기술적 깊이를 판단할 수 있는 3~4개를 추린다.

각 주제는 이 흐름을 따른다:

```
문제 (무엇이 안 되고 있었는가)
  → 접근 (어떻게 어프로치했는가, 왜 그 방식을 선택했는가)
  → 난관 (과정에서 부딪힌 기술적 문제)
  → 해결 (구체적으로 어떻게 풀었는가)
  → 결과 (수치로)
```

**주제 선택 기준:**
- 본인의 설계 판단이 명확히 드러나는 것
- 수치로 결과를 증명할 수 있는 것
- 기술적 난이도 또는 비자명한(non-obvious) 해결이 있는 것

### 4. 근거: 수치, 코드, 다이어그램

```
# 금지
"메모리 누수를 해결하여 안정성이 향상되었다."

# 대신
"인쇄당 42-62MB 누적 → 4-5회 후 OOM. try/finally + Destroy()로 누수 제거.
 연속 인쇄 횟수 제한 해소."
```

mermaid 다이어그램, 코드 스니펫, 비교 테이블을 적극 사용한다.

### 5. 마케팅/감상 금지

```
# 금지
"이 프로젝트를 통해 성장했다"
"도전적인 과제였지만 극복했다"
"많은 것을 배울 수 있었다"
"혁신적인 아키텍처를 설계했다"
```

이런 문장은 어떤 정보도 전달하지 않는다. 삭제하고 구체적 사실로 대체한다.

### 6. 기술 스택: 역할 기반 서술

```
# 금지
기술 스택: C#, Unity, HLSL, Python, Supabase

# 대신
| 기술 | 적용 영역 |
|------|-----------|
| C# / Unity 6 | 13개 런타임 시스템, 디바이스 추상화, 3단계 초기화 |
| HLSL | 4-pass 컬러 보정 파이프라인 (CommandBuffer) |
| P/Invoke | 프린터·카메라 등 6종 네이티브 SDK 통합 |
```

## Quick Reference

| 요소 | 규칙 |
|------|------|
| 톤 | 간결 서술체 + 체언 종결 혼합. 자연스러운 리듬 |
| 첫 장 | 임팩트 퍼스트 — 문제 → 해결 → 기여/결과 수치 |
| 주제 | 3~4개 임팩트 중심으로 추려서 |

| 관점 | 라벨 없이 서술 속에 자연스럽게. depth 있는 철학 |
| 근거 | 수치, 코드, 다이어그램 |
| 감상 | 절대 금지. "배웠다", "성장했다", "깨달았다" 사용 불가 |

## 검증 체크리스트

작성 완료 후 반드시 아래 항목을 전부 점검:

1. **첫 장에 임팩트가 있는가?** — 프로젝트 소개 → 문제 → 해결 → 기여/결과가 첫 페이지에서 보이는가
2. **"사견:", "관점:" 같은 명시적 라벨이 없는가?** — 관점은 판단 근거를 서술하는 흐름 속에서 자연스럽게 드러나야 한다
3. **"배웠다", "성장했다", "깨달았다", "도전적이었다" 같은 뻔한 감상이 없는가?** — 관점은 구체적 경험에서 도출된 기술적 원칙이어야 한다
4. **모든 주장에 수치/현상 근거가 있는가?** — "개선되었다", "향상되었다" 같은 형용사 퉁치기 없이
5. **주제가 3~4개로 추려져 있는가?** — 전체 시스템 나열이 아닌 임팩트 중심 선별
6. **각 주제가 "문제 → 접근 → 난관 → 해결 → 결과" 흐름을 따르는가?** — 구현만 나열하지 않고, 왜 이렇게 판단했는가가 드러나야 한다
7. **마케팅 언어가 없는가?** — "혁신적", "최적의", "뛰어난" 등 전부 삭제

## Common Mistakes

1. **블로그 톤 혼입** — "~했다" 1인칭 서술이나 "사견:" 라벨. 포트폴리오는 간결 서술체.
2. **전체 나열** — 13개 시스템을 전부 설명. 임팩트 있는 3~4개만 추린다.
3. **감상 삽입** — "많은 걸 배웠다", "도전적이었다". 구체적 사실로 대체.
4. **구현만 서술** — "무엇을 했다"만 씀. "왜 이렇게 판단했는가"와 "무엇이 어려웠는가"가 핵심.
5. **결과 누락** — 설계 판단만 쓰고 수치 결과를 안 씀. 항상 결과로 닫는다.
6. **한 가지 어미만 반복** — "~적용. ~설계. ~구현." 같은 기계적 체언 종결 반복. 서술체와 섞어서 리듬을 만든다.
